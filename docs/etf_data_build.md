# ETF 周线数据构建文档

## 概述

本文档记录从 Tushare Pro 拉取 ETF 行情数据，聚合成周线 CSV 的完整流程、技术决策、踩坑记录。

**目标文件**: `data/all_etfs_nav_2013_<最后日期>.csv`
**辅助脚本**: `scripts/generate_etf_weekly_qfq.py`
**数据时间范围**: 2013-01-04 ~ 最新交易日
**快照频率**: 周线，每周最后交易日（优先周五）
**价格类型**: 前复权（Forward Adjusted）

## 2. 数据拉取

### 2.1 Tushare Token

```python
import tushare as ts
ts.set_token('YOUR_TOKEN')
pro = ts.pro_api()
```

### 2.2 接口选择

| 数据类型 | 接口 | 参数 |
|---------|------|------|
| ETF 日线 | `pro.fund_daily()` | `ts_code='513100.SH'` |
| 指数日线 | `pro.index_daily()` | `ts_code='H20269.CSI'` |
| 复权因子 | `pro.fund_adj()` | `ts_code='512890.SH'` |

### 2.3 分段拉取

为避免单次请求数据量过大，按年份分段拉取后合并：

```python
for year in range(2013, 2027):
    df = pro.fund_daily(ts_code='513100.SH',
                        start_date=f'{year}0101',
                        end_date=f'{year}1231')
    dfs.append(df)
result = pd.concat(dfs, ignore_index=True)
```

## 3. 前复权处理

### 3.1 为什么需要前复权

ETF 在运作过程中会发生**基金拆分**和**分红**，导致价格出现非自然的跳变：

| ETF | 事件类型 | 日期 | 复权因子变化 |
|:---|:---|:---:|:---:|
| **512890** | **基金拆分** | **2021-10-25** | **1.0 → 2.0462**（实际拆分比约 1:2） |
| **513100** | **基金拆分** | **2022-01-14** | **1.0 → 5.0019**（实际拆分比约 1:5） |
| 510500 | 多次分红 | 上市以来 | 0.2803 → 0.3341（累计） |
| 518880 | 无 | — | 恒为 1.0 |
| 511010 | 多次分红 | 2025~ | 1.0121 → 1.0375（累计） |

不复权数据在拆分日会出现 ~50% 或 ~80% 的虚假跳变，必须使用前复权。

### 3.2 前复权计算公式

```python
# 前复权价格 = 不复权价格 × 当日复权因子 / 最新复权因子
close_qfq = close * adj_factor_t / latest_adj_factor
```

例如 512890 在 2019-01-18（拆分前）：
- 不复权 = 1.003，当日因子 = 1.0，最新因子 = 2.0462
- 前复权 = 1.003 × 1.0 / 2.0462 = **0.4902**

> ⚠️ **坑**: Tushare 的 `pro_bar(adj='qfq')` 对 ETF 类型 (`asset='FD'`) 不生效，返回的数据与 `fund_daily` 完全一致。必须手动通过 `fund_adj` API 获取复权因子自行计算。

### 3.3 同花顺 vs Tushare 复权差异

| 平台 | 拆分因子 | 2019-01-25 前复权 | 含义 |
|:---|:---:|:---:|:---|
| 同花顺 | 2.0 | 0.506 | **仅拆分复权**，未含分红 |
| Tushare | **2.0462** | **0.4941** | **拆分 + 分红全复权** ✅ |

Tushare 的复权因子包含了基金拆分和累计分红，更准确。同花顺等行情软件可能只做拆分复权，导致历史价格偏高约 2.3%，收益率被低估。

## 4. 红利低波ETF 特殊处理

### 4.1 背景

512890（红利低波ETF）于 **2019-01-18** 才上市，而策略回测需要从 2013 年开始的数据。

解决方案：使用**中证红利低波全收益指数 H20269.CSI** 填充 2013-2018 年的数据。

### 4.2 踩坑：固定比例缩放法（❌ 错误）

**最初的做法**：计算 H20269 与 512890 在重叠期间（2019年1月）的固定比例，然后用这个比例缩放 H20269 的绝对值。

```python
# ❌ 错误做法
ratio = median(512890_close / H20269_close)  # 约 0.000044
H20269_scaled = H20269_raw × ratio
```

**问题**：H20269 的价格水平（~10000 点）与 512890（~1.0 元）差了上万倍，固定缩放因子会引入数值误差，导致收益率不准确。

### 4.3 正确方法：收益率反推法（✅）

**核心思想**：H20269 是全收益指数，其**收益率曲线**是准确的。用收益率反推价格，而非用固定比例缩放。

```python
# ✅ 正确做法
# 基准点：512890 首个交易日(2019-01-18)
base_price = 0.4902   # 512890 前复权价
base_index = 11046.32  # H20269 当日收盘

# 反推任意日期的历史价格
P_t = base_price × (H20269_t / H20269_base)

# 示例：2013-05-17
P_20130517 = 0.4902 × (5612.62 / 11046.32) = 0.2491
```

**优点**：
- ✅ 收益率 = H20269 真实总收益率（含分红）
- ✅ 价格水平 = 512890 前复权价格水平
- ✅ 2019-01-18 前后无缝衔接
- ✅ 与旧文件（h20269_scaled）差异仅在于缩放锚点不同

### 4.4 旧文件差异说明

原有文件 `all_etfs_nav_2013_2026_h20269_scaled.csv` 的缩放锚点不同，导致同样的 H20269 收益率被缩放到不同的价格水平：

| 指标 | 旧文件 | 新文件 |
|:---|:---:|:---:|
| 2019-01 锚点 | 0.5239（手工设定） | 0.4902（512890前复权） |
| 缩放因子 | 0.00004743 | 0.00004438（反推法） |
| 2013-05-17 值 | 0.3145 | 0.2491 |
| 总收益率(红利低波) | +276.45% | +390.25% |

旧文件收益率偏低（+276% vs +390%），因为其缩放锚点与 512890 实际价格不匹配。

## 5. 周线聚合

### 5.1 快照日：从周一改为周五

原有数据使用**每周一**作为快照日。为了更方便手工维护，改为**每周最后交易日**（优先周五）：

```python
for (year, week), group in daily_data.groupby(['isoyear', 'isoweek']):
    overall_last_date = group['trade_date'].max()
    row = {'trade_date': overall_last_date}
    for etf in ETF_NAMES:
        etf_data = group.dropna(subset=[etf])
        if len(etf_data) > 0:
            last_etf = etf_data.sort_values('trade_date').iloc[-1]
            row[etf] = last_etf[etf]
        else:
            row[etf] = np.nan
```

### 5.2 踩坑：日期对齐

> ⚠️ **坑**: 最初周线聚合时，对所有 ETF 取统一的最后交易日。但当某 ETF 在周五暂停交易（如 512890 在 2021-10-22 除权除息日停牌），会导致该列出现 NaN。

**修复**：每个 ETF 独立取该周的最后有效交易日，行日期取所有 ETF 的最后交易日。

## 6. 日期合并边界问题

### 6.1 踩坑：H20269 与 512890 的分界点

> ⚠️ **坑**: 最初以 `2019-01-01` 为分界，< 2019 用 H20269，≥ 2019 用 512890。但 512890 直到 **2019-01-18** 才上市，导致 2019-01-04 和 2019-01-11 两周的数据丢失。

**修复**：分界点改为 512890 的实际上市日期 `2019-01-18`。

## 7. 数据质量检查

每次生成后自动执行：

1. **空值检查** — 各列 NaN 数量及占比
2. **日期连续性** — 周间隔是否超过 14 天（春节、国庆等假期正常）
3. **拆分前后验证** — 512890 在 2021-10 拆分前后价格是否连续
4. **最新值核对** — 与 Tushare `fund_daily` 最新收盘价对比
5. **收益率一致性** — 红利低波反推部分与 H20269 收益率一致

## 8. 手动维护指南

### 8.1 追加每周数据

每周五收盘后，直接在 CSV 末尾追加一行：

```csv
2026-07-03,2.2000,1.0900,4.9500,8.5000,140.8000
```

价格来源：Tushare `fund_daily(ts_code='XXXX.SH', trade_date='20260703')` 的 `close` 字段。

### 8.2 全量重跑

```bash
cd /home/ubuntu/claw_etf_strategy
python3 scripts/generate_etf_weekly_qfq.py
```

输出文件：`data/all_etfs_nav_2013_<日期>.csv`
软链接：`data/all_etfs_nav_latest.csv` → 最新文件

### 8.3 注意事项

- 追加的价格必须是 **Tushare 不复权收盘价**（脚本会自动计算前复权，但一次性全量跑时才做）
- 手动追加时沿用已有文件的数值风格
- 当 512890 发生新的拆分/分红时，需重新全量跑脚本
