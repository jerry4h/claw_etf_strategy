【数据清洗与交易日历处理】（重要技术细节，影响实操）

> 以下问题在2026年4月全量搜索期间被发现，当前搜索结果已包含此偏差（系统性偏差，不影响参数相对排序）。
> **下一轮精搜前必须修复**，否则波动率因子将持续被低估，策略可能偏向过高风险标的。

### 1. 原始数据的日期特征

原始CSV（`all_etfs_nav_2013_2026_merged.csv`）中：
- **周末**：全部为 **NaN**（共1,348天）
- **节假日（工作日休市）**：全部为 **NaN**（共231天，含春节、国庆、清明等调休）
- **交易日**：有真实净值数据

### 2. 当前代码的问题

`data_loader.py` 第20行的 `ffill()` 把**所有NaN**（包括周末和节假日）全部填充为前一日价格：

```python
# 当前（错误）：周末/节假日被ffill填充成0收益日
df[col] = df[col].ffill()
```

**后果：**
- 20日动量窗口实际只有约**14个交易日**（被6个0收益日稀释）
- 20日波动率被系统性**低估约2-3个百分点**（0波动日拉低了标准差）
- 年化乘数 `√252` 与实际交易日不匹配

**2024年春节典型案例：**
```
2月8日（周四） 1.308 ← 真实数据
2月9日-18日    NaN   ← 春节+周末休市
2月19日（周一） 1.301 ← 真实数据

ffill后 → 2月9日到2月18日连续11天 = 1.308（幽灵交易日）
```

### 3. 正确修复方案

```python
def load_nav_data():
    df = pd.read_csv(DATA_PATH)
    df["日期"] = pd.to_datetime(df["日期"])
    df = df.set_index("日期").sort_index()

    # Step 1: 删除"全市场休市"的行（所有ETF都是NaN）
    # 这自然包含了周末和A股节假日，无需外部节假日日历
    all_nan = df[ETFS].isna().all(axis=1)
    df = df[~all_nan].copy()

    # Step 2: 单只ETF的个别缺失值（如QDII暂停申赎），用ffill处理
    for col in ETFS:
        if col in df.columns:
            df[col] = df[col].replace("", np.nan)
            df[col] = df[col].ffill()

    df = df.dropna(how="all")
    return df
```

### 4. 对回测引擎的影响

| 影响项 | 修复前 | 修复后 |
|--------|--------|--------|
| 日频数据条数 | 4,721（含幽灵日） | ~3,242（纯交易日） |
| 20日动量窗口 | ~14个交易日 | **20个交易日** |
| 20日波动率 | 低估2-3% | **无偏估计** |
| 年化乘数 | `√252` 不匹配日历日窗口 | `√252` 匹配真实交易日 |
| 周频调仓点 | 675个（含62个被填充的节假日周一） | ~674个 |

**对 `resample("W-MON")` 的影响：**
- 修复后，原本62个"被ffill填充的节假日周一"会消失
- pandas 会自动将该周最后一个真实交易日（通常是周五）作为该周的锚点
- **调仓从"周一"顺延到"周五"**，但周频不变，对策略逻辑无实质影响

### 5. 对实操的指导

**手工计算时如何处理？**
- 周一查净值时，如果某天是节假日（A股休市），直接跳过，等开市后再算
- 不需要自己维护节假日日历——策略是**周频调仓**，每周只操作一次，节假日只要跨过去即可
- 波动率计算如果手工做，用**20个交易日的收盘价**（不是20个日历日）

**为什么 `√252` 是对的？**
- 252 ≈ 365天 - 104个周末 - 约9个节假日
- A股实际交易日约244-246天，但量化业界统一使用 `√252` 作为跨市场可比标准
- 与 `√244` 的差异仅 **1.6%**，对策略排序影响可忽略
